# Data Migration Guide

Complete guide for migrating production data to preview/local environments.

## Overview

Two-step process to export production data and import it into preview or local environments:

1. **Export** - Download sanitized data from production
2. **Import** - Upload data to preview/local environment

## Features

✅ **Selective Migration** - 100 users, 100 vehicles, max 2 images per vehicle  
✅ **Data Sanitization** - Emails and phone numbers replaced with safe placeholders  
✅ **Reusable** - Exported data can be used for multiple environments  
✅ **Storage Included** - Downloads and uploads images from Supabase Storage  
✅ **Complete Relationships** - All foreign keys and relationships preserved  
✅ **Schema Version Independent** - Uses raw SQL, works with any schema version

## Step 1: Export from Production

### Prerequisites

- Access to production database
- Access to production Supabase Storage
- Sufficient disk space (~500MB-2GB depending on images)

### Environment Variables

Create a `.env.production` file with standard environment variable names:

```bash
# .env.production
# Auto-generated by Supabase CLI

# Database URLs
DATABASE_URL="postgresql://postgres.[PROJECT]:[PASSWORD]@aws-0-us-east-1.pooler.supabase.com:6543/postgres?pgbouncer=true&connection_limit=1"
DIRECT_URL="postgresql://postgres:[PASSWORD]@db.[PROJECT].supabase.co:5432/postgres"

# Supabase
NEXT_PUBLIC_SUPABASE_URL="https://[PROJECT].supabase.co"
NEXT_PUBLIC_SUPABASE_ANON_KEY="[ANON_KEY]"
SUPABASE_SERVICE_ROLE_KEY="[SERVICE_ROLE_KEY]"
```

### Run Export

```bash
# Load production env vars and run export
env $(grep -v '^#' .env.production | grep -v '^$' | xargs) npm run export-production-data
```

### What Gets Exported

**All Lookup Tables:**
- ✅ Countries
- ✅ Steering Types
- ✅ Makes & Models
- ✅ Collections

**User Data (100 users):**
- ✅ Top 100 users by vehicle count
- ✅ Email: `user-[N]-[hash]@preview.spokehire.com`
- ✅ Phone: `+44 7700 [NNNN]`
- ✅ All other profile data preserved

**Vehicle Data (100 vehicles):**
- ✅ Most recent 100 vehicles from selected users
- ✅ All vehicle specifications
- ✅ All vehicle collections
- ✅ All vehicle sources
- ✅ Status, visibility, etc.

**Media (max 2 images per vehicle):**
- ✅ Prioritizes primary images
- ✅ Only `READY` status images
- ✅ Downloads from Supabase Storage
- ✅ Stores locally for reuse

**Deals (20 most recent):**
- ✅ Active deals only
- ✅ Deal vehicles
- ✅ Deal recipients

### Output

```
data/migration/
├── exported-data.json     # All database records (JSON)
└── images/                # Downloaded images
    ├── [mediaId]_[filename].jpg
    ├── ...
```

**File sizes:**
- `exported-data.json`: ~2-5 MB
- `images/`: ~100-500 MB (depends on image sizes)

## Step 2: Import to Preview/Local

### For Preview Environment

```bash
# Load preview environment variables
export $(grep -v '^#' .env.preview | grep -v '^$' | xargs)

# Run import
npm run import-preview-data
```

### For Local Environment

```bash
# Make sure local Supabase is running
npm run supabase:start

# Run import with --local flag
npm run import-preview-data -- --local
```

### What Happens During Import

1. **Verification** - Checks if data file exists
2. **Database Check** - Shows existing data counts
3. **Lookup Tables** - Imports/updates all reference data
4. **Users** - Imports with sanitized emails/phones
5. **Vehicles** - Imports with all specifications
6. **Relationships** - Restores all vehicle collections, sources, etc.
7. **Images** - Uploads to Supabase Storage bucket
8. **Media Records** - Links images to vehicles
9. **Deals** - Imports deals with recipients
10. **Verification** - Confirms all counts match

### Import Behavior

- **Upsert Mode** - Existing records are updated (by ID)
- **Safe** - Won't delete existing data
- **Idempotent** - Can be run multiple times safely

# 1. Setup database

## Configuration

### Adjust Export Limits

Edit `scripts/export-production-data.ts`:

```typescript
const CONFIG = {
  MAX_USERS: 100,              // Change to export more/fewer users
  MAX_VEHICLES: 100,           // Change to export more/fewer vehicles
  MAX_IMAGES_PER_VEHICLE: 2,   // Change to export more/fewer images
  MAX_DEALS: 20,               // Change to export more/fewer deals
  // ...
};
```

### Customize Sanitization

Email format in `sanitizeEmail()`:
```typescript
function sanitizeEmail(originalEmail: string, index: number): string {
  const hash = crypto.createHash('md5').update(originalEmail).digest('hex').substring(0, 8);
  return `user-${index}-${hash}@preview.spokehire.com`;
}
```

Phone format in `sanitizePhone()`:
```typescript
function sanitizePhone(originalPhone: string | null, index: number): string | null {
  if (!originalPhone) return null;
  const lastFour = String(index).padStart(4, '0');
  return `+44 7700 ${lastFour}`; // UK format
}
```

## Troubleshooting

### Schema version mismatch errors

The export script uses **raw SQL queries** to be schema-version independent. It will export whatever columns exist in the source database, regardless of your local Prisma schema.

This means:
- ✅ Works if production has older schema than local
- ✅ Works if production has newer schema than local  
- ✅ No need to sync schemas before exporting

### Export fails with "DATABASE_URL not found"

Make sure you're loading the environment variables correctly:
```bash
env $(grep -v '^#' .env.production | grep -v '^$' | xargs) npm run export-production-data
```

Or create a `.env.production` file with:
```bash
DATABASE_URL="postgresql://..."
NEXT_PUBLIC_SUPABASE_URL="https://..."
SUPABASE_SERVICE_ROLE_KEY="..."
```

### Import fails with "Export file not found"

Run the export step first:
```bash
npm run export-production-data
```

### Images fail to download

- Check production Supabase credentials
- Verify `SUPABASE_SERVICE_ROLE_KEY` has storage access
- Check network connectivity

### Images fail to upload

- Check target environment has `vehicle-images` bucket
- Verify Supabase service role key
- Run `npm run db:setup` to create bucket

### Database counts don't match

Some records may fail due to:
- Foreign key constraints (if lookup data is missing)
- Duplicate key constraints (if running import twice)
- Check error logs in console output

## Performance

### Export Time
- Small dataset (100 vehicles): ~2-5 minutes
- Large dataset (500 vehicles): ~10-15 minutes
- Bottleneck: Image downloads from Supabase Storage

### Import Time
- Preview (cloud): ~3-7 minutes
- Local: ~2-5 minutes
- Bottleneck: Image uploads to Supabase Storage

### Storage Requirements
- JSON data: ~2-5 MB
- Images (2 per vehicle × 100): ~100-500 MB
- Total: ~102-505 MB per export

## Automation (Optional)

### GitHub Actions Example

```yaml
name: Refresh Preview Data
on:
  schedule:
    - cron: '0 0 * * 0'  # Weekly on Sunday
  workflow_dispatch:

jobs:
  refresh:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      
      - name: Export from production
        env:
          DATABASE_URL: ${{ secrets.PROD_DB_URL }}
          DIRECT_URL: ${{ secrets.PROD_DIRECT_URL }}
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.PROD_SUPABASE_URL }}
          SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.PROD_SERVICE_KEY }}
        run: npm run export-production-data
      
      - name: Import to preview
        env:
          DATABASE_URL: ${{ secrets.PREVIEW_DB_URL }}
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.PREVIEW_SUPABASE_URL }}
          SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.PREVIEW_SERVICE_KEY }}
        run: npm run import-preview-data
```
